# Planning a change in this repo

You are in the **plan** phase of the PIV workflow. Your only output is an approved plan file under `.claude/plans/`. Implementation and verification belong to `piv-implement` and `piv-validate` — do not encroach on them, even when the change looks trivial. Separating the phases is the whole point: a one-line "obvious" fix that turns out to be wrong is exactly what this workflow exists to catch.

Follow these steps in order.

## 1. Restate the task

In one sentence, paraphrase what the user wants. This catches misunderstandings before they become a wasted plan. Wait for confirmation only if your restatement materially differs from what they said.

## 2. Investigate lightly

Read or grep only:

- Files and symbols the user named explicitly.
- Files those references obviously imply (e.g., user names a controller — read its route group; user names a component — read its parent route file).

Do NOT:

- Spawn Explore, general-purpose, or other subagents.
- Take open-ended tours of the codebase.
- Read whole packages "to get a feel."

If the task genuinely needs broader investigation, surface that under **Open questions** and ask the user — do not go fishing. Heavy investigation is the user's call to authorize, not yours to assume.

## 3. Know the repo's shape

Use these facts when reasoning about affected areas. Verify with a quick `Read` or `Glob` if a fact looks load-bearing for your plan — the repo evolves.

- `packages/api` — .NET 10 minimal API. EF Core (Sqlite), Scalar for OpenAPI. Entry point `Program.cs`.
- `packages/api.tests` — unit tests against `api` (uses `InternalsVisibleTo`).
- `packages/api.tests.integration` — integration tests against `api`.
- `packages/web` — TanStack Start (React 19, Vite). Tailwind 4, Radix UI, Zod 4. Path alias `#/*` → `./src/*`. API client via `openapi-fetch` against generated types in `src/api/schema.ts`.
- **API ↔ Web type contract**: `packages/web/src/api/schema.ts` is generated by `bun gen:api`, which runs `npx openapi-typescript http://localhost:5204/openapi/v1.json -o ./src/api/schema.ts` against a *running* API on port 5204. Any plan that changes API request/response shapes MUST include an explicit step "regenerate web types via `bun gen:api` in `packages/web`" and note that the API needs to be running locally for the regen to succeed.

## 4. Draft the plan

Use this exact section structure. Keep it terse — bullets over prose, paths over descriptions.

```
# <task title>

## Goal
<one or two sentences — the user-facing outcome>

## Affected areas
- packages/<pkg>/<path/to/file.ext>
- ...

## Approach
1. <step>
2. <step>
...

## Out of scope
- <something this plan deliberately doesn't cover>

## Open questions
- <ambiguity the user needs to resolve>
- (write "None" if there are none)

## Validation strategy
- <what piv-validate should check>
```

Section guidance:

- **Affected areas** lists file paths only. If you don't know a path, say so under Open questions — don't guess.
- **Approach** is a numbered ordered list. Each step should be small enough that piv-implement can do it without re-planning. If a step would itself need a plan, split it.
- **Validation strategy** is the contract you hand to piv-validate. Be concrete: name commands and files, e.g. `dotnet test packages/api.tests`, `bun gen:api && bun check` in `packages/web`, "manually load /foo and confirm X renders".

## 5. Persist the plan

Write the plan to `.claude/plans/<slug>.md`:

- `<slug>` is kebab-case, derived from the task, max ~6 words (e.g. `add-user-favorites`, `fix-openapi-nullability`).
- Create `.claude/plans/` if it doesn't exist.
- If a file with that slug already exists, append `-2`, `-3`, etc. Never silently overwrite — a stale plan on disk is better than a lost one.

After writing, tell the user the absolute path of the file. piv-implement will read from that path.

## 6. Present for approval

Call `ExitPlanMode` with the plan content so the user can approve, edit, or reject.

- If they approve: stop. The next phase is `piv-implement`, which will read the plan file.
- If they ask for edits: update the file on disk first, then re-present via `ExitPlanMode`. The file is the source of truth — don't carry an edited version only in conversation.
- If they reject the approach entirely: discuss, then redraft from step 1.

## 7. Hard constraints

These rules are absolute. Violating them defeats the point of separating plan from implement.

- **Never edit source code.** Even a one-line "obvious" fix belongs to piv-implement. The only files you write are the plan file under `.claude/plans/`.
- **Never run tests, builds, formatters, type-checkers, or any verification commands.** No `dotnet test`, no `bun check`, no `bun gen:api`, no `bun build`. If you think a check is needed, list it under Validation strategy for piv-validate to run.
- **Never spawn background agents** (`run_in_background: true`). Keep all work in the foreground so the user can interrupt and so progress is visible. Foreground subagents are also discouraged in this phase — light investigation should be doable inline with Read and Grep.
